流水线（pipeline）

以较简单的 MIPS 架构为例，可以将一条指令的处理过程分为取指（IF）、译码（ID）、执行（EX）、访存（MEM）、写回（WB）​五个阶段，每个阶段由不同的电路处理。

可见，在处理一条指令的一个阶段时，其他电路处于空闲状态，资源利用率很低。

而流水线技术通过把指令交叠起来提高效率，同时处理多条指令的不同阶段。

超标量（superscalar）

超标量 CPU 架构是指在单个处理器内核中实现了指令级并行（与多线程不同）​的一类并行运算，可以分派多条指令在不同的执行单元中执行，充分利用处理器内核中的多个执行单元。

乱序执行（out-of-order execution, OOE）

为了提升整体效率，CPU 不一定会按照汇编代码中指令的顺序执行。

乱序执行的步骤通常是将指令译码后放入重排序缓冲区（re-order buffer, ROB），同时也放入保留站（reservation station, RS）。当 RS 中的一条指令可以执行时（依赖的数据已经计算完毕且有可用的执行资源），会被分配到一个合适的执行单元中执行。

而指令执行完毕后结果会写入该指令在 ROB 中的对应位置，表面上看这个指令还并没有执行，因为它没有修改数据。最后当该指令前的指令都将结果写回后，这条指令的结果才会写入寄存器中。

这样就做到了​“乱序执行，顺序提交”​，从外部看来指令仍然是顺序执行的。

寄存器重命名（register renaming）

对于 WAR 和 WAW 两种依赖，可以使用寄存器重命名技术解决。

CPU 中物理寄存器的数量（physical register）远多于架构寄存器（architectural register）的数量。

所以对于这两条指令，可以将 (3) 的结果放入另一个物理寄存器中，这样虽然指令 (2) 和指令 (3) 都使用了 r2，但实际上使用的是不同的物理寄存器，也就消除了 WAR 型依赖。

WAW 型依赖也是同样的方法处理。

但在现代 CPU 中，因为指令乱序执行，并且需要使用转发，所以并不会简单地记录每个架构寄存器对应的物理寄存器，而是用 ROB 代替了物理寄存器，用寄存器别名表（register alias table, RAT）记录 ROB 中最后一次修改某个架构寄存器的指令。

控制冒险（control hazards）

当 CPU 流水线中遇到了条件跳转指令时，不能在开始时就判断出分支结果，就会导致控制冒险。

加气泡是万能方法，但效率极低，流水线越长性能损失越大。

高效的解决方法是使用分支预测（branch prediction），选一条分支开始执行，如果预测错误，则恢复到分支指令执行完毕时刻的状态。

因为分支预测错误的代价很高，所以预测准确率就极其重要。

静态分支预测

最简单的分支预测就是单方向静态预测，直接预测所有分支都跳转/不跳转。

略复杂一点的方法是预测向后的分支会跳转，而向前的分支不会跳转，因为往往向后的跳转是一个循环结构。

动态分支预测

动态预测利用分支指令历史跳转情况进行预测，准确率比静态预测高很多，现在几乎所有处理器都使用动态预测。

饱和计数器（saturating counter）

饱和计数器是一种较简单的动态分支预测方法，用 n 个二进制位表示一个无符号整数，初值为 0，如果分支跳转了就让数值 +1，如果不跳转就让数值 -1，当操作会导致溢出时则不进行操作。预测时根据最高位的值进行预测。

两级自适应预测器（two-level adaptive predictor）

如果一个分支是隔一次跳转一次，那么虽然有很明显的规律，但饱和计数器却无法预测。

而两级自适应预测器解决了这个问题。这是目前最广泛最成熟的预测方法，可以快速学会预测一定周期内的任意重复模式。

这种预测器的预测方法是首先建立一个模式历史表（pattern history table），包括 \(2^k\) 个饱和计数器，每次预测时，将该分支的最近 k 次跳转记录用状态压缩的方法表示为一个 \(0\sim2^k-1\) 的整数，然后找到对应的饱和计数器进行预测。

比如 \(k=4\) 时，如果前 4 次执行该指令的情况分别为不跳转、跳转、不跳转、跳转，那么可以状态压缩为 \((0101)_2\)，即 5，然后用第 5 个饱和计数器进行预测。

条件传送指令

从 Pentium Pro 开始，Intel 加入了一系列条件传送指令（例如 cmove 表示相等时传送），使用这些指令代替简单的分支，在数据不可预测时，可以大大提高执行速度。

但当数据有序时，可能会比分支略慢，这在较早的一些架构的 CPU 上尤为明显，因为这些架构上条件传送指令的执行周期较长。但从 Broadwell 开始，条件传送指令的执行速度提高了，那么就不会比分支慢多少了。

骆可强的 WC2009 论文《论程序底层优化的一些方法与技巧》中提到：
